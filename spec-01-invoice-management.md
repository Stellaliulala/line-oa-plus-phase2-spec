**版本**：1.2 | **日期**：2026-05-21 | **依據文件**：DX Solution Phase 2 Product Guide § 14.2

---

## 1. 功能概述

發票由 12CM（服務提供商）負責開立，OA Plus 平台不維護發票資料。使用者在 12CM 系統內填寫發票資訊，12CM 系統須獨立承接發票資料管理與開票流程。

支援兩種發票類型：**電子發票**（個人）與**三聯式統一發票**（公司行號）。不支援捐贈發票，載具不支援捐贈選項。

**整合方式：導流至 12CM 外部頁面**

LINE 採用「連結跳轉」方式，而非在 LINE 頁面內嵌填寫。付款成功後 LINE 顯示連結按鈕，使用者點擊後另開新視窗跳轉至 12CM 開發的發票填寫頁。訂閱管理頁面亦提供連結，供使用者日後更新發票資訊（如更換載具）。

## 2. 責任範圍

| 範疇 | 負責方 | 說明 |
|------|--------|------|
| 發票開立 | 12CM | 由 12CM 系統產生與發送發票 |
| 發票資訊收集 | 12CM | 使用者在 12CM 系統填寫抬頭、統編等資訊 |
| 付款成功事件觸發 | OA Plus 平台 | 透過 `PAYMENT_SUCCEEDED` Webhook 通知 12CM |
| 付款快照查詢 | 12CM | 收到 Webhook 後呼叫 Open API 取得付款明細 |
| 待開票紀錄管理 | 12CM | 發票資料未齊全時，建立待開票紀錄，資料補齊後補開 |

## 3. 發票類型定義

### 3.1 電子發票（個人）

使用者選擇個人抬頭時適用。

| 欄位 | 必填 | 格式 / 說明 |
|------|------|------------|
| 抬頭類型 | Y | 固定為「個人」 |
| 電子信箱 | Y | 發票寄送用，需通過 email 格式驗證 |
| 載具類型 | Y | 二選一：Email 寄送（會員載具）、手機條碼 |
| 手機條碼 | 條件必填 | 選擇「手機條碼」時必填，格式：`/XXXXXXX`（斜線 + 7 碼英數大寫） |
| 聯絡電話 | N | 選填 |

### 3.2 三聯式統一發票（公司）

使用者選擇公司抬頭時適用，同時停用載具設定。

| 欄位 | 必填 | 格式 / 說明 |
|------|------|------------|
| 抬頭類型 | Y | 固定為「公司」 |
| 公司抬頭 | Y | 公司全名，不可為空 |
| 統一編號 | Y | 8 位純數字，需通過格式驗證 |
| 電子信箱 | Y | 發票寄送用 |
| 聯絡電話 | N | 選填 |

## 4. 開票流程

### 4.1 正常流程（付款前已填妥發票資訊）

```
使用者完成付款
    │
    ▼
OA Plus 發送 PAYMENT_SUCCEEDED Webhook
    │
    ▼
12CM 接收 Webhook（驗證 X-DX-Signature）
    │
    ▼
12CM 呼叫 Retrieve Payment API 取得付款快照
    │
    ▼
12CM 查詢該 merchantId 是否已有完整發票資料
    │
  資料齊全
    │
    ▼
觸發開票，發票寄送至使用者 Email
```

### 4.2 延遲開票流程（付款後才補填發票資訊）

```
使用者完成付款，但尚未填寫發票資訊
    │
    ▼
OA Plus 發送 PAYMENT_SUCCEEDED Webhook
    │
    ▼
12CM 接收 Webhook，查詢該 merchantId 發票資料
    │
  資料不齊全（欄位缺漏或從未填寫）
    │
    ▼
12CM 建立待開票紀錄（status: PENDING）
    ├── 儲存 paymentOrderId、merchantId、付款金額、付款時間
    └── 狀態標記為 PENDING
    │
    ▼
Admin Plugin 顯示補填提示 banner，引導使用者至發票填寫頁
    │
    ▼
使用者完成發票資訊填寫並送出
    │
    ▼
12CM 觸發補開流程
    ├── 查詢該 merchantId 所有 status = PENDING 的待開票紀錄
    └── 依序補開發票，成功後更新 status 為 ISSUED
    │
    ▼
發票寄送至使用者 Email
```

## 5. 待開票紀錄（`invoice_pending`）資料結構

> 以下為建議資料結構，需由工程師依實際系統設計調整。

| 欄位 | 類型 | 說明 |
|------|------|------|
| `id` | string | 待開票紀錄唯一 ID |
| `merchantId` | string | 商戶識別碼 |
| `paymentOrderId` | string | 對應的 OA Plus 付款訂單 ID |
| `amount` | number | 付款金額（來自 Retrieve Payment API） |
| `paidAt` | timestamp | 付款完成時間 |
| `status` | enum | `PENDING`：待補填 / `ISSUED`：已開立 / `FAILED`：開立失敗 |
| `createdAt` | timestamp | 本筆紀錄建立時間 |
| `issuedAt` | timestamp | 發票開立時間（ISSUED 後填入） |

## 6. 發票資訊填寫頁面規格

### 6.1 表單欄位與驗證規則

| 欄位 | 必填 | 驗證規則 | 錯誤提示文案 |
|------|------|---------|------------|
| 抬頭類型 | Y | 單選（個人 / 公司） | — |
| 公司抬頭 | 公司必填 | 非空字串 | 請輸入公司抬頭 |
| 統一編號 | 公司必填 | `/^\d{8}$/` | 請輸入正確的 8 位統一編號 |
| 電子信箱 | Y | `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` | 請輸入有效的電子信箱 |
| 載具類型 | 個人必填 | 單選（Email / 手機條碼） | — |
| 手機條碼 | 條件必填 | `/^\/[A-Z0-9]{7}$/` | 格式應為 /XXXXXXX |
| 聯絡電話 | N | — | — |

### 6.2 UI 互動邏輯

- 抬頭類型切換為「公司」：隱藏載具設定，顯示公司抬頭與統一編號欄位
- 抬頭類型切換為「個人」：隱藏公司資訊區塊，顯示載具設定
- 載具選擇「手機條碼」：展開條碼輸入欄位
- 送出時進行全欄位驗證，未通過則於對應欄位下方即時顯示錯誤訊息
- 儲存成功後顯示 success banner，3 秒後自動消失

### 6.3 補填提示 Banner

若使用者有 `PENDING` 狀態的待開票紀錄，Admin Plugin 頁面頂部應顯示：

> 您有 N 筆付款尚未完成發票資訊填寫，請前往填寫，填寫完成後系統將自動補開發票。

### 6.4 進入發票填寫頁的路徑

使用者進入 12CM 發票填寫頁面有三條路徑，均透過 LINE 端的**連結按鈕跳轉**至 12CM 外部頁面（另開新視窗），不在 LINE 頁面內嵌填寫：

1. **路徑一 — 付款成功頁面**：使用者完成藍新付款後，LINE 付款成功頁面顯示「前往填寫發票資訊」連結按鈕，點擊後跳轉至 12CM 填寫頁。LINE 建議將填寫動作置於付款完成後，避免流程斷點影響轉化率。
2. **路徑二 — Admin Plugin 補填 Banner**：付款後若未填寫，系統建立 PENDING 紀錄，Admin Plugin 頂部顯示補填提示 Banner，點擊連結跳轉至 12CM 填寫頁。
3. **路徑三 — 訂閱管理頁面**：OA Plus 訂閱管理頁面提供連結，使用者可在下一期扣款前跳轉至 12CM 頁面更新發票資訊（如更換載具、修改統編）。

### 6.5 開立紀錄規格

發票管理頁需呈現每筆已開立或處理中的發票紀錄：

| 欄位 | 說明 |
|------|------|
| 付款日期 | 對應 `paidAt` |
| 付款金額 | 對應 `amount` |
| 發票狀態 | `PENDING` / `ISSUED` / `FAILED` |
| 發票號碼 | 開立成功後顯示（ISSUED） |
| 開立日期 | 對應 `issuedAt`（ISSUED 後顯示） |
| 操作 | FAILED 狀態提供「重新開立」按鈕 |

## 6.8 商戶註冊頁 — 發票專區

商戶首次啟用 12CM 服務時（不論免費版或付費版），須完成註冊頁填寫。發票相關欄位集中於註冊頁的「發票專區」（為註冊頁內的獨立分區，非獨立頁面），填寫後資料將自動帶入發票管理頁，無需重複填寫。

### 6.8.1 完整欄位規格

**基本資料**

| 欄位 | 必填 | 驗證規則 | 備註 |
|------|------|---------|------|
| 品牌名稱 | Y | 中英文符號皆可，不允許 emoji | 12CM 內部識別用 |
| 品牌代號 | Y | 英文字母（大小寫）、數字、符號，不允許 emoji；送出前檢查是否重複，重複回傳「代號已存在，請重新輸入」 | 唯一值 |

| 聯絡人姓名 | Y | 非空字串 | |
| 聯絡人電話 | N | — | 選填 |
| 聯絡人 Email | Y | email 格式 | |

| 欄位 | 必填 | 驗證規則 | 備註 |
|------|------|---------|------|
| 抬頭類型 | Y | 單選：個人 / 公司 | |

| 手機條碼 | 條件必填 | `/^\/[A-Z0-9]{7}$/` | 選擇「手機條碼」時必填 |

### 6.8.2 UI 互動邏輯

- 抬頭類型切換為「個人」：顯示載具類型、手機條碼（條件顯示）、發票寄送 Email；隱藏公司發票欄位
- 抬頭類型切換為「公司」：顯示買受人名稱（預設帶入公司名稱）、統一編號、買受人地址、買受人電話、發票寄送 Email；隱藏載具設定
- 發票寄送 Email 預設帶入聯絡人 Email，使用者可自行修改
- 品牌代號失焦時即時呼叫 API 檢查是否重複
- 送出時進行全欄位驗證

### 6.8.3 與發票管理頁的資料關係

註冊頁填寫的發票專區資料，將自動帶入發票管理頁（§6）的發票資訊設定 tab，商戶無需重複填寫。後續如需變更，可至發票管理頁或透過 LINE 訂閱管理頁的連結進行更新。

## 7. Webhook 接收規格

### 7.1 事件類型

使用 `PAYMENT_SUCCEEDED` 事件作為開票流程觸發點。

### 7.2 Payload 欄位

```json
{
  "eventType": "PAYMENT_SUCCEEDED",
  "eventId": "evt-002",
  "pluginId": "testplugin",
  "merchantId": "@merchant123",
  "paymentOrderId": "pay-001",
  "paymentStatus": "SUCCESS",
  "sentAt": "2026-03-30T16:40:00Z"
}
```

| 欄位 | 類型 | 必填 | 說明 |
|------|------|------|------|
| `eventType` | string | Y | 固定為 `PAYMENT_SUCCEEDED` |
| `eventId` | string | Y | 此事件唯一識別碼 |
| `pluginId` | string | Y | Plugin 識別碼 |
| `merchantId` | string | Y | 商戶識別碼 |
| `paymentOrderId` | string | Y | 付款訂單 ID，用於查詢明細 |
| `paymentStatus` | string | Y | 例如 `SUCCESS` |
| `sentAt` | Timestamp | Y | ISO 8601 UTC |

### 7.3 Signature 驗證

```
X-DX-Signature: sha256={HMAC-SHA256 of raw request body}
```

12CM 必須用 **Secret of Admin Plugin** 驗證簽章：

```java
String expectedSig = "sha256=" + hmacSha256(rawBody, secret);
if (!MessageDigest.isEqual(expectedSig.getBytes(), receivedSig.getBytes())) {
    throw new UnauthorizedException("Invalid signature");
}
```

收到 Webhook 後，12CM 需呼叫 **Retrieve Payment API** 取得完整付款快照。

| 項目 | 值 |

| 環境 | Staging only（目前） |
| 用途 | 取得付款快照資料，作為後續開票與帳務處理依據 |

## 9. Webhook 回應規範

| 回應狀態 | 平台行為 |
|---------|---------|
| 2xx | 成功，不重試 |
| 4xx | 永久失敗，不重試 |
| 5xx / 429 | 暫時失敗，最多重試 3 次（指數退避：1s → 2s → 4s） |

---

## 10. 注意事項

- Webhook 僅作為事件通知，不包含完整付款明細，**必須**另行呼叫 Retrieve Payment API
- 同一筆 `paymentOrderId` 只能對應一張發票，處理前需檢查是否已存在 `ISSUED` 或 `PENDING` 紀錄，避免重複建立
- 補開發票時需依序處理所有 `PENDING` 紀錄，若其中一筆失敗（`FAILED`），不影響其他筆繼續處理
- 發票開立失敗須記錄失敗原因，並提供人工介入或重試機制
- 手機條碼格式在前端即時驗證；若日後需串接財政部 API 做二次驗證，可在後端補充
- 前端 Prototype（填寫頁、管理頁）詳見 HTML 規格文件內嵌互動元件
