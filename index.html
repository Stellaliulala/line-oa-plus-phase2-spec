1 功能概述

發票由 12CM（服務提供商）負責開立，OA Plus 平台不維護發票資料。使用者在 12CM 系統內填寫發票資訊，12CM 系統須獨立承接發票資料管理與開票流程。

整合方式：導流至 12CM 外部頁面
LINE 採用「連結跳轉」而非頁面內嵌方式。付款成功後 LINE 在頁面上顯示連結按鈕，使用者點擊後另開新視窗跳轉至 12CM 開發的發票填寫頁。訂閱管理頁面亦提供連結，供使用者日後更新發票資訊（如更換載具）。

支援兩種發票類型：

  • 電子發票（個人抬頭）
  • 三聯式統一發票（公司行號）

發票開立頻率

幫手市集採訂閱制，首次付款成功後即啟動訂閱，往後每月同一天自動續扣。每次續扣成功，OA Plus 均會發送一次 PAYMENT_SUCCEEDED Webhook，12CM 每次收到此事件都應觸發一次開票流程。

工程注意：開票並非一次性行為。每位商戶每月均會產生一筆新的開票紀錄，DB 設計需支援一對多關係（一個 merchantId 對應多筆 invoice_pending 紀錄），發票號碼管理也須考量每月累積的資料量。
付款事件觸發來源開票行為
首次訂閱付款使用者完成訂閱 Checkout觸發第一次開票流程
每月自動續扣OA Plus 平台自動執行每月同一天觸發一次開票流程
寬限期補救扣款成功OA Plus 平台 retry同樣觸發開票流程

4 開票流程

4.1 正常流程（付款前已填妥發票資訊）

1
使用者完成付款
2
OA Plus 發送 PAYMENT_SUCCEEDED Webhook LINE 規格
3
12CM 接收 Webhook，驗證 X-DX-Signature
4
12CM 呼叫 Retrieve Payment API 取得付款快照
5
12CM 查詢該 merchantId 是否已有完整發票資料 內部邏輯
6
資料齊全 → 觸發開票，發票寄送至使用者 Email

4.2 延遲開票流程（付款後才補填發票資訊）

1
使用者完成付款，但尚未填寫發票資訊
2
OA Plus 發送 PAYMENT_SUCCEEDED Webhook LINE 規格
3
12CM 查詢 merchantId 發票資料 → 資料不齊全
4
12CM 建立待開票紀錄（status: PENDING）內部邏輯
儲存 paymentOrderId、merchantId、付款金額、付款時間
狀態標記為 PENDING
5
Admin Plugin 顯示補填提示 banner，引導使用者至發票填寫頁
6
使用者完成發票資訊填寫並送出
7
12CM 觸發補開流程 內部邏輯
查詢該 merchantId 所有 status = PENDING 紀錄
依序補開發票，成功後更新 status 為 ISSUED
8
發票寄送至使用者 Email

6.5 開立紀錄規格

發票管理頁的「開立紀錄」Tab 呈現該商戶所有付款對應的發票開立狀況，工程師須依據以下規格實作列表欄位、狀態邏輯與顯示規則。

欄位定義

欄位名稱資料來源必填顯示規則
付款日期invoice_pending.paidAtY格式：YYYY-MM-DD，依付款時間降冪排列
付款金額invoice_pending.amountY格式：NT$XXX
發票狀態invoice_pending.statusY見下方狀態說明
發票號碼開票系統回傳NISSUED 後顯示號碼；PENDING / FAILED 顯示「—」
開立日期invoice_pending.issuedAtNISSUED 後顯示日期；PENDING / FAILED 顯示「—」

發票狀態定義

狀態值顯示文字顯示樣式說明
PENDING待補填黃底深橘字發票資料尚未填寫，等待使用者補填後觸發開票
ISSUED已開立綠底深綠字發票已成功開立，發票號碼與開立日期同步顯示
FAILED開立失敗紅底深紅字開票過程發生錯誤，需人工介入處理

排序與分頁規則

  • 預設依 paidAt 降冪排列（最新付款在最上方）
  • 建議每頁顯示 10 筆，超過時提供分頁或無限捲動（待工程確認實作方式）
  • PENDING 狀態的紀錄優先置頂，提醒使用者盡快補填

補填後的即時更新行為

  • 使用者在「發票資訊設定」Tab 送出後，系統觸發補開流程
  • 補開完成後，列表中對應的 PENDING 紀錄應即時更新為 ISSUED，並填入發票號碼與開立日期
  • 若補開失敗，狀態更新為 FAILED，需人工介入或系統排程重試
  • 更新方式：補開 API 回應後前端 re-fetch 列表，或由後端 push 狀態變更（待工程決定實作方式）
注意：同一筆 paymentOrderId 只能對應一張發票。補開前須確認該筆紀錄狀態為 FAILED，避免重複開票。

X-DX-Signature: sha256={HMAC-SHA256 of raw request body}

12CM 必須用 Secret of Admin Plugin 驗證簽章：

String expectedSig = "sha256=" + hmacSha256(rawBody, secret);

    throw new UnauthorizedException("Invalid signature");

8.2 12CM 發票開立 API

內部邏輯：12CM 擁有自有的發票開立 API 與字軌管理機制，無需依賴第三方發票服務。本節說明開票流程的關鍵決策。

開票金額來源

使用 Retrieve Payment API 回傳的付款金額，該金額為含稅金額，直接用於開票，不需另行計算稅額。

開票觸發時機

使用者填寫完發票資訊送出後立即觸發開票 API，不採用非同步排程。

開票失敗處理

情境處理方式使用者介面
開票 API 失敗 每 3 分鐘 retry 一次，寫入告警報告 不顯示失敗狀態
Email 寄送失敗 寫入告警報告 不顯示失敗狀態

告警報告供工程師排查用，不對外呈現。

6.8 商戶註冊頁 — 發票專區

功能概述

12CM 是藍新金流的收款方（Merchant），須完成藍新金流帳號申請與綁定，才能在 OA Plus 上支援付費 Plugin。

金流操作（授權、請款、退款、自動續扣、retry）全部由 OA Plus 平台代為執行，12CM 不直接呼叫藍新 API。12CM 的角色是透過 Webhook 與 Open API 取得付款結果，並據此處理內部權益開通與發票流程。

ℹ️

付費版啟用成功判斷條件 內部邏輯

收到 ACTIVATION_CHANGED 並查詢後，須滿足以下全部條件才視為進階版啟用成功：

✓
activatedStatus = ACTIVATED
✓
pluginLevel = PREMIUM
✓
Subscription subscriptionStatus = active 或 trialing
ℹ️
三個條件必須同時成立，任一條件不符則不應開通進階功能。

Trial 期間的判斷：subscriptionStatus = trialing 時視同 active，進階版有效。Trial 到期後首次扣款若失敗，進入寬限期（主狀態轉為 ACTIVATED），最終 retry 仍失敗則 EXPIRED，降回免費版。
📌
Trial 授權成功後 activatedStatus 的實際值待確認 待確認 Product Guide §8.2 說明「Subscription 進入 TRIALING」，但未明確說明此時 activatedStatus 的值。若 Trial 期間 activatedStatus 不是 ACTIVATED，上述第一個條件的判斷邏輯需調整。實作前請向 LINE 技術窗口確認。
!

寬限期（Grace Period）說明 LINE 規格

當自動續扣或 Trial 到期後的首次正式扣款失敗時，LINE 平台不會立即將訂閱標記為失效，而是進入寬限期。

項目說明
寬限期長度約 7–10 天，由 LINE 平台決定，12CM 無需設定
期間訂閱狀態ACTIVATED（維持不變，使用者仍可使用付費功能）
retry 機制由 LINE 平台自動執行，12CM 無需介入
retry 成功訂閱恢復正常，LINE 發送 PAYMENT_SUCCEEDED
寬限期結束仍失敗訂閱轉為 EXPIRED，LINE 發送 ACTIVATION_CHANGED，12CM 據此停止付費功能
ℹ️
12CM 不需主動判斷寬限期 寬限期的進入與結束均由 LINE 平台管理。12CM 只需監聽 Webhook：收到 ACTIVATION_CHANGED 且狀態為 EXPIRED 時才停止付費功能，期間維持原狀即可。

CSRF 安全機制 LINE 規格

所有非 GET 請求（POST / PUT / DELETE）必須帶入 CSRF Token：

const csrfToken      = dx.security.getCSRFToken();
const csrfHeaderName = dx.security.getCSRFTokenHeaderName(); // "DX-CSRF-TOKEN"

fetch('/api/your-endpoint', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    [csrfHeaderName]: csrfToken
  },
  body: JSON.stringify(payload)
});
⚠️
缺少 CSRF Token 的非 GET 請求將被拒絕。每次操作前都需重新取得最新 Token，不應快取舊 Token。

DX-JWT 後端驗證 LINE 規格

OA Plus 反向代理會在每個請求的 cookie 中注入 dx-jwt，12CM 後端須驗證此 JWT。

JWT 結構

Header
{ "alg": "HS256", "typ": "JWT" }
Payload
{
  "pluginId": "12345",
  "merchantId": "123456",
  "pluginInfo": {
    "level": "FREE",
    "activatedStatus": "ACTIVATED"
  },
  "iat": 1717023000,
  "exp": 1717026600
  // role 欄位是否存在待 LINE 技術窗口確認
}

驗簽方式

使用 LINE 提供的 Secret of Admin Plugin 以 HMAC-SHA256 驗簽。

項目說明
演算法HMAC-SHA256
Secret 來源LINE 核發的 Secret of Admin Plugin（詳見 Phase 2 對接清單）
Token 有效期60 分鐘（exp 欄位）
注入方式反向代理自動注入至 dx-jwt cookie
