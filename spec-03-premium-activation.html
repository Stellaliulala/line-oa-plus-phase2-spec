1

功能概述

當使用者完成付費訂閱後，12CM 系統須根據 OA Plus 平台回傳的 Subscription 狀態與 Plugin Level，開通對應的付費功能權益（進階版），並在訂閱失效時即時停止相關功能存取。

ℹ️
本文件範疇 涵蓋：訂閱生命週期完整流程（含 Trial）、狀態模型定義、Webhook 處理機制、API 查詢方式、前後端權益控管邏輯、進階版啟用判斷條件，以及開發測試方式。
2

商戶訂閱完整生命週期流程 內部邏輯

下圖呈現商戶從訂閱起點到各種終止情境的完整狀態流轉，包含 Trial 路徑、正式扣款、每月自動續扣、失敗寬限期與降回免費版三條收斂路徑。

📌
依會議紀錄確認，Trial 期間為 1 個月（30 天），由 LINE 平台設定。流程圖中原以「N 天」標示，現已確定為 30 天。
⚠️
寬限期機制確認（LINE 規格） Trial 到期後扣款失敗時，適用 LINE 平台的寬限期機制，並非直接降為免費版。寬限期內系統仍處於 ACTIVATED 狀態。

訂閱狀態模型

3.1 核心狀態欄位 LINE 規格

欄位可能值說明
activatedStatus ACTIVATED
ACTIVATION_INITIATED
ACTIVATION_FAILED 		服務是否已啟用
pluginLevel FREE / PREMIUM 目前使用的方案層級
subscriptionStatus active / trialing / inactive Subscription 是否有效。trialing 為 Trial 期間專屬狀態，視同 active，進階版功能有效

3.2 狀態轉換說明 內部邏輯

觸發事件activatedStatus 變化pluginLevel 變化備註
使用者發起訂閱 → ACTIVATION_INITIATED 維持 FREE 等待付款結果
Trial 授權成功 待確認 Subscription → TRIALING
activatedStatus 實際值待向 LINE 技術窗口確認		 → PREMIUM 進階版立即啟用，Trial 期間（30 天）視同訂閱中
Trial 到期首次扣款成功 → ACTIVATED 維持 PREMIUM 正式訂閱開始，收到 PAYMENT_SUCCEEDED
付款成功（無 Trial） → ACTIVATED → PREMIUM 進階版啟用
付款失敗（無 Trial） → ACTIVATION_FAILED 維持 FREE 使用者需重新訂閱
Trial 到期首次扣款失敗 → ACTIVATED（寬限期）→ EXPIRED → FREE（寬限期結束後） 進入寬限期（主狀態為 ACTIVATED），retry 仍失敗後降回免費版
寬限期 retry 仍失敗 → EXPIRED → FREE 功能立即停止
使用者取消（到期後） → EXPIRED → FREE 本期結束前仍 ACTIVATED

Webhook 事件處理

4.1 ACTIVATION_CHANGED LINE 規格

觸發時機：activation 狀態或 plugin level 發生變更時。

Payload 範例

{
  "eventType": "ACTIVATION_CHANGED",
  "eventId": "evt-001",
  "pluginId": "testplugin",
  "merchantId": "@merchant123",
  "activatedStatusChanged": {
    "previous": "INACTIVATED",
    "current": "ACTIVATED"
  },
  "pluginLevelChange": {
    "previous": "FREE",
    "current": "PREMIUM"
  },
  "updatedTime": "2026-03-30T16:40:00Z",
  "sentAt": "2026-03-30T16:40:00Z"
}

12CM 處理步驟 內部邏輯

  1. 驗證 X-DX-Signature
  2. 呼叫 Retrieve Subscription V2 API 取得最新 Subscription 狀態與 activation 詳細資訊
  3. 依查詢結果同步 12CM 系統中的服務權益
⚠️
重要：Webhook 僅為通知，不含完整資料 Webhook 僅作為事件通知，不直接提供完整訂閱明細，必須另行呼叫 Retrieve Subscription V2 API 查詢最新狀態。

權益控管邏輯 內部邏輯

ℹ️
12CM 的角色：被動接收，不主動追蹤 12CM 不需主動查詢訂閱狀態，只需根據 LINE 平台透過 Webhook 推送的事件同步調整系統可用版本。用量等級變更、付費版啟用、降級、寬限期進入與到期，均由 LINE 平台判斷後通知。

12CM 須同時判斷三個維度：

OA Paid / Unpaid  ×  OA 用量等級（輕用量 / 中高用量） ×  DX Subscription 狀態（active / inactive）

6.1 用量等級說明 內部邏輯

LINE OA 帳號分為輕用量與中高用量兩種等級（Phase 1 既有概念），會影響 12CM 系統的可用功能範圍。用量等級變更由 LINE 透過 Webhook 通知，12CM 已有對應處理。

用量等級免費版功能範圍備註
中高用量 全功能開放 正常免費版
輕用量 優惠券不可新增/編輯，其餘功能正常 Phase 1 限制，若升級付費版則跳過此限制

6.2 完整權益判斷矩陣 內部邏輯

OA 付費狀態OA 用量等級DX Subscription12CM 系統狀態
Paid 中高用量 Active（PREMIUM） ✅ 完整付費功能
Paid 輕用量 Active（PREMIUM） ✅ 完整付費功能（跳過輕用量限制）
Paid 中高用量 Inactive / EXPIRED 降回正常免費版（全功能可用）
Paid 輕用量 Inactive / EXPIRED ⚠️ 回到 Phase 1 限制（優惠券不可新增/編輯）並顯示輕用量升級提示
Unpaid 任何 任何 ⛔ 限制功能並提示升級 OA 付費方案
⚠️
輕用量用戶升級付費版後享有完整功能。訂閱到期或取消且本期結束後，系統須判斷其用量等級：若仍為輕用量，需回到 Phase 1 限制狀態並顯示輕用量升級提示，而非降回一般免費版。

6.3 前端（Storefront / MINI App）

🛑
subscription_status = inactive 時，建議暫停或隱藏 Storefront，避免使用者下新訂單。

6.4 後端（CMS / Admin Plugin）

  • Subscription inactive 時限制商戶僅可查看既有訂單，停用需有效方案才能操作的功能
  • 輕用量且 Subscription inactive 時，額外限制優惠券新增/編輯，並顯示輕用量升級提示（沿用 Phase 1 既有實作）

6.5 API 層 LINE 規格

  • 停止請求新的 Auth Token（API 將直接回失敗）
  • 停止發送 Open API 事件（OA Plus 將封鎖請求）
7

付費版啟用成功判斷條件 內部邏輯

收到 ACTIVATION_CHANGED 並查詢後，須滿足以下全部條件才視為進階版啟用成功：

✓
activatedStatus = ACTIVATED
✓
pluginLevel = PREMIUM
✓
Subscription subscriptionStatus = active 或 trialing
ℹ️
三個條件必須同時成立，任一條件不符則不應開通進階功能。

Trial 期間的判斷：subscriptionStatus = trialing 時視同 active，進階版有效。Trial 到期後首次扣款若失敗，進入寬限期（主狀態轉為 ACTIVATED），最終 retry 仍失敗則 EXPIRED，降回免費版。
📌
Trial 授權成功後 activatedStatus 的實際值待確認 待確認 Product Guide §8.2 說明「Subscription 進入 TRIALING」，但未明確說明此時 activatedStatus 的值。若 Trial 期間 activatedStatus 不是 ACTIVATED，上述第一個條件的判斷邏輯需調整。實作前請向 LINE 技術窗口確認。
!

寬限期（Grace Period）說明 LINE 規格

當自動續扣或 Trial 到期後的首次正式扣款失敗時，LINE 平台不會立即將訂閱標記為失效，而是進入寬限期。

項目說明
寬限期長度約 7–10 天，由 LINE 平台決定，12CM 無需設定
期間訂閱狀態ACTIVATED（維持不變，使用者仍可使用付費功能）
retry 機制由 LINE 平台自動執行，12CM 無需介入
retry 成功訂閱恢復正常，LINE 發送 PAYMENT_SUCCEEDED
寬限期結束仍失敗訂閱轉為 EXPIRED，LINE 發送 ACTIVATION_CHANGED，12CM 據此停止付費功能
ℹ️
12CM 不需主動判斷寬限期 寬限期的進入與結束均由 LINE 平台管理。12CM 只需監聽 Webhook：收到 ACTIVATION_CHANGED 且狀態為 EXPIRED 時才停止付費功能，期間維持原狀即可。
8

Idempotency 處理 內部邏輯

12CM 發送事件（如 createOrder）時，須帶入唯一 eventId 作為 idempotency key，防止重複操作。

建議格式
{merchantId}:{orderId}:{eventType}:{timestamp}
範例
abc123:12345:Buy:1752122552
📌
以下為建議資料結構，需由工程師依實際系統設計調整。timestamp 建議使用 Unix epoch（秒），確保全域唯一性。

錯誤處理 內部邏輯

情境HTTP 狀態建議處理
API 回傳 429（Rate Limit） 429 實作指數退避重試（Exponential Backoff）
Webhook 驗簽失敗 401 回傳 401，記錄日誌，不處理事件
查詢 API 失敗 5xx / timeout 保留當前狀態，排入重試佇列
所有重試失敗 — 告警通知，人工確認

開發與測試 LINE 規格

10.1 測試帳號

LINE 將提供兩種方案等級的測試帳號：

帳號類型 A
Light

驗證免費版行為及功能限制情境。

帳號類型 B
Medium / High

驗證付費版（PREMIUM）行為與升降級情境。

10.2 Development Mode

在 Staging 環境（https://oaplus-staging.line.biz/）可啟用 Developer Mode，透過 OA Plus 右下角浮動按鈕切換測試目標 URL，便於開發驗證。

⚠️
環境限制 Development Mode 目前僅支援 RC（Staging）環境，Production 不支援。
