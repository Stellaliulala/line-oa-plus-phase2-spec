1 功能概述

發票由 12CM（服務提供商）負責開立，OA Plus 平台不維護發票資料。使用者在 12CM 系統內填寫發票資訊，12CM 系統須獨立承接發票資料管理與開票流程。

整合方式：導流至 12CM 外部頁面
LINE 採用「連結跳轉」而非頁面內嵌方式。付款成功後 LINE 在頁面上顯示連結按鈕，使用者點擊後另開新視窗跳轉至 12CM 開發的發票填寫頁。訂閱管理頁面亦提供連結，供使用者日後更新發票資訊（如更換載具）。

支援兩種發票類型：

  • 電子發票（個人抬頭）
  • 三聯式統一發票（公司行號）

發票開立頻率

幫手市集採訂閱制，首次付款成功後即啟動訂閱，往後每月同一天自動續扣。每次續扣成功，OA Plus 均會發送一次 PAYMENT_SUCCEEDED Webhook，12CM 每次收到此事件都應觸發一次開票流程。

工程注意：開票並非一次性行為。每位商戶每月均會產生一筆新的開票紀錄，DB 設計需支援一對多關係（一個 merchantId 對應多筆 invoice_pending 紀錄），發票號碼管理也須考量每月累積的資料量。
付款事件觸發來源開票行為
首次訂閱付款使用者完成訂閱 Checkout觸發第一次開票流程
每月自動續扣OA Plus 平台自動執行每月同一天觸發一次開票流程
寬限期補救扣款成功OA Plus 平台 retry同樣觸發開票流程

4 開票流程

4.1 正常流程（付款前已填妥發票資訊）

1
使用者完成付款
2
OA Plus 發送 PAYMENT_SUCCEEDED Webhook LINE 規格
3
12CM 接收 Webhook，驗證 X-DX-Signature
4
12CM 呼叫 Retrieve Payment API 取得付款快照
5
12CM 查詢該 merchantId 是否已有完整發票資料 內部邏輯
6
資料齊全 → 觸發開票，發票寄送至使用者 Email

4.2 延遲開票流程（付款後才補填發票資訊）

1
使用者完成付款，但尚未填寫發票資訊
2
OA Plus 發送 PAYMENT_SUCCEEDED Webhook LINE 規格
3
12CM 查詢 merchantId 發票資料 → 資料不齊全
4
12CM 建立待開票紀錄（status: PENDING）內部邏輯
儲存 paymentOrderId、merchantId、付款金額、付款時間
狀態標記為 PENDING
5
Admin Plugin 顯示補填提示 banner，引導使用者至發票填寫頁
6
使用者完成發票資訊填寫並送出
7
12CM 觸發補開流程 內部邏輯
查詢該 merchantId 所有 status = PENDING 紀錄
依序補開發票，成功後更新 status 為 ISSUED
8
發票寄送至使用者 Email

6.5 開立紀錄規格

發票管理頁的「開立紀錄」Tab 呈現該商戶所有付款對應的發票開立狀況，工程師須依據以下規格實作列表欄位、狀態邏輯與顯示規則。

欄位定義

欄位名稱資料來源必填顯示規則
付款日期invoice_pending.paidAtY格式：YYYY-MM-DD，依付款時間降冪排列
付款金額invoice_pending.amountY格式：NT$XXX
發票狀態invoice_pending.statusY見下方狀態說明
發票號碼開票系統回傳NISSUED 後顯示號碼；PENDING / FAILED 顯示「—」
開立日期invoice_pending.issuedAtNISSUED 後顯示日期；PENDING / FAILED 顯示「—」

發票狀態定義

狀態值顯示文字顯示樣式說明
PENDING待補填黃底深橘字發票資料尚未填寫，等待使用者補填後觸發開票
ISSUED已開立綠底深綠字發票已成功開立，發票號碼與開立日期同步顯示
FAILED開立失敗紅底深紅字開票過程發生錯誤，需人工介入處理

排序與分頁規則

  • 預設依 paidAt 降冪排列（最新付款在最上方）
  • 建議每頁顯示 10 筆，超過時提供分頁或無限捲動（待工程確認實作方式）
  • PENDING 狀態的紀錄優先置頂，提醒使用者盡快補填

補填後的即時更新行為

  • 使用者在「發票資訊設定」Tab 送出後，系統觸發補開流程
  • 補開完成後，列表中對應的 PENDING 紀錄應即時更新為 ISSUED，並填入發票號碼與開立日期
  • 若補開失敗，狀態更新為 FAILED，需人工介入或系統排程重試
  • 更新方式：補開 API 回應後前端 re-fetch 列表，或由後端 push 狀態變更（待工程決定實作方式）
注意：同一筆 paymentOrderId 只能對應一張發票。補開前須確認該筆紀錄狀態為 FAILED，避免重複開票。

8.2 12CM 發票開立 API

內部邏輯：12CM 擁有自有的發票開立 API 與字軌管理機制，無需依賴第三方發票服務。本節說明開票流程的關鍵決策。

開票金額來源

使用 Retrieve Payment API 回傳的付款金額，該金額為含稅金額，直接用於開票，不需另行計算稅額。

開票觸發時機

使用者填寫完發票資訊送出後立即觸發開票 API，不採用非同步排程。

開票失敗處理

情境處理方式使用者介面
開票 API 失敗 每 3 分鐘 retry 一次，寫入告警報告 不顯示失敗狀態
Email 寄送失敗 寫入告警報告 不顯示失敗狀態

告警報告供工程師排查用，不對外呈現。

6.8 商戶註冊頁 — 發票專區

商戶首次啟用 12CM 服務（不論免費版或付費版）時，須完成註冊頁填寫。發票相關欄位集中於註冊頁的「發票專區」，填寫後資料將自動帶入發票管理頁，無需重複填寫。

內部邏輯：發票專區為註冊頁的獨立分區，不是獨立頁面。資料填寫後與發票管理頁（§6）共用同一份資料。

6.8.1 抬頭類型選擇

商戶可能以個人或公司名義付款，因此抬頭類型需在註冊時明確選擇，並依選擇顯示對應欄位。

6.8.4 UI 互動邏輯

  • 抬頭類型切換為「個人」：顯示載具類型、手機條碼（條件顯示）、發票寄送 Email；隱藏公司發票欄位

  • 發票寄送 Email 預設帶入聯絡人 Email，使用者可自行修改
  • 載具選擇「手機條碼」：展開手機條碼輸入欄位
  • 品牌代號失焦時即時呼叫 API 檢查是否重複，重複時顯示錯誤訊息「代號已存在，請重新輸入」
  • 送出時進行全欄位驗證，未通過則於對應欄位下方即時顯示錯誤訊息

6.8.4 與發票管理頁的資料關係

✅
註冊頁填寫的發票專區資料，將自動帶入發票管理頁（§6）的發票資訊設定 tab，商戶無需重複填寫。後續如需變更（如更換載具、修改統編），可至發票管理頁或透過 LINE 訂閱管理頁的連結進行更新。

9 Webhook 回應規範 LINE 規格

回應狀態平台行為
2xx成功，不重試
4xx永久失敗，不重試
5xx / 429暫時失敗，最多重試 3 次（指數退避：1s → 2s → 4s）

10 注意事項

  • Webhook 僅作為事件通知，不包含完整付款明細，必須另行呼叫 Retrieve Payment API LINE 規格
  • 同一筆 paymentOrderId 只能對應一張發票，處理前需檢查是否已存在 ISSUED 或 PENDING 紀錄，避免重複建立 內部邏輯
  • 補開發票時需依序處理所有 PENDING 紀錄，若其中一筆失敗（FAILED），不影響其他筆繼續處理 內部邏輯
  • 發票開立失敗須記錄失敗原因，並提供人工介入或重試機制 內部邏輯
  • 手機條碼格式在前端即時驗證；若日後需串接財政部 API 做二次驗證，可在後端補充
  • 開票失敗與 Email 寄送失敗均不在使用者介面顯示，統一寫入告警報告供工程師排查
